Project Repo

每个 Concordia 项目的知识空间。不是被动的文件夹，而是有明确流转逻辑的系统——信息从外部进入，经过 AI 加工和人类审批，变成项目的共享认知，最终产出交付物。

六个模块

state/
项目的大脑——认知状态、信号指令、成员立场

整个系统最核心的部分。PROJECT.md 是项目的宪法，signal-mandate.md 定义该关注什么，cognition/ 子系统追踪项目的共享认知和成员理解。

cognition/ 是 Concordia 最有防御性的层——平台能做摘要，但"知道谁的认知和项目真相不一致"是独有能力。

shared-state.md

项目共识。经 DRI 确认的项目真相索引。"项目现在认为是真的东西。"

objects/

认知对象（目标/决策/风险/假设）。每个由 signal 支撑，经 DRI 确认。candidate → accepted

signals/

信号是证据，不是结论。AI 发现的候选信号经 DRI 判断后准入，成为认知对象的支撑。

stances/

成员立场。每人一个文件，AI 推断或本人确认的对项目的理解。inferred → aligned

gaps.md

认知差距。自动对比 shared-state 和每个 stance，列出谁的理解和项目真相不一致。

Project Agent + DRI 校准 实时更新
sources/
项目的输入素材——经人工审批后才入库

所有进入项目的参考资料。重要规则：素材不会自动入库——source-collection 扫描后产出候选报告存到 _pending/，经 source-review 和用户确认后才写入正式目录。

按来源分三个子目录：内部沟通（owner-global）、SharePoint 文档、公开网络。Member Agent 入库到 owner-global/{member-alias}/ 子文件夹。

source-collection 产出 source-review 审批 每天 2 次
drafts/
AI 草稿区——等人 review 后搬到 deliverables

AI 心跳自动产出的工作草稿暂存在这里。按成员分子文件夹。草稿不是正式交付物——需要人 review 批准后才搬到 deliverables/。

解决的问题：AI 不用等人来了才开始工作，先做完存草稿，人只需要审批成品。

Heartbeat 自动产出 每小时
deliverables/
正式交付物——经人工审批后归档

经人工审批后的正式产出物。agent-tasks/ 存 Project Agent 派发的任务交付物（验收后移到 done/），tasks/ 存成员主动提交的交付物。

成员提交 DRI review 按需
logs/
自动化日志——只追加不修改

回答"过去发生了什么"。每个文件创建后不再修改，形成完整的项目运行轨迹。heartbeats/ 记录每小时巡检，reviews/ 记录每日 review 决策，decisions/ 记录重大决策。

自动写入 每小时 / 每天
learnings/
经验教训——每周反思后写入

项目执行过程中积累的可复用经验。采用 Zettelkasten 卡片式管理——每条经验独立编号，包含原则（一句话规则）+ 场景（怎么学到的）+ 关联（相关的其他经验）。

Agent 从心跳日志和 review 记录中找 pattern，和 DRI 确认后写入。一周的信息量够发现 pattern，太早沉淀容易写出"正确但无用"的经验。

Agent 提炼 DRI 确认 每周

设计原则

收集不入库

AI 扫描产出候选报告存到 _pending/，不自动写入 sources/。入库需要人类明确批准。

AI 先亮模型

认知对象预填 candidate，成员立场预填 inferred。人类是校准者不是空白作者。

扫描全面 呈现精炼

7 个信源全覆盖，但给用户看的是按 topic 归类的认知判断，不是 raw signal dump。

Signal 是证据 Object 是结论

signals/ 存外部事件，objects/ 存项目认为是真的事。从证据到结论需要 DRI 确认。

沉默不等于同意

成员没回复认知校准问题，标记 unknown，不标 aligned。Agent 不能把推断静默变成项目真相。

Planner 是任务唯一数据源

不维护 todo-list.md 影子副本。Planner 任务是项目任务的 single source of truth。
