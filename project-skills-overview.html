Concordia Project Skills

从项目启动到日常运行，9 个 Skill 构成完整的项目认知与执行循环。PMO Agent 管全局，Member Agent 管个人，DRI 做校准。

首次运行

项目启动时顺序执行一次，建立基础设施和初始认知
1
Project Init
建项目 + 创建基础设施

和用户对话收集项目目标、交付物、成功标准、deadline、团队成员。确认后一键创建 Teams team + channel、Planner plan（Open / In Progress / Done）、SharePoint Project Repo（6 模块目录结构）、Project Agent + workspace。

对话从 Stage G（帮用户想清楚要做什么）开始，不直接问项目名——从目标反推名字更自然。

用户说"创建项目" 引导式对话 Teams + Planner + SharePoint + Agent
2
Signal Mandate
定义"项目该关注什么"

8 个阶段的引导式对话，帮 DRI 定义项目的信息监控策略：关注方向、信源范围、价值标准、升级阈值、隐私边界、频率时效。

核心原则：不接受 TBD，每个维度必须有明确结论。AI 先提建议让 DRI 校准，不问空白问题。产出的 signal-mandate.md 是后续所有收集和扫描的行动纲领。

Init 完成后触发 8 阶段对话 signal-mandate.md
3
First Collection & Review
首次全量收集 + 认知对齐 + 入库审批

Phase 1 — 收集（自动）：8 个子 agent 并行扫描 Teams 私聊/群聊/会议群聊、邮件、会议纪要、日历、SharePoint（回溯 6 个月）、Web 公开搜索。每条候选信息带可点击链接、内容摘要、AI 推断。

Phase 2 — 认知对齐（交互）：将多条信号合并为认知判断，逐个和 DRI 做 Truth Gate。确认后更新 cognition 层（objects / stances / gaps）。

Phase 3 — 入库审批（交互）：按类别呈现候选素材，DRI 批量或逐条 approve 后写入 sources/。完成后引导用户进入任务规划。

Signal Mandate 后触发 Phase 2+3 需要 DRI cognition/ + sources/
4
Planning
深度任务规划 + 建议分工

读 PROJECT.md + 已入库的 sources/ + cognition 状态，先做自我审计（知识够不够？逻辑完整吗？可行吗？），再拆解成依赖排序的任务计划。

产出包含：规划思路（为什么选这条路）、任务列表 + 建议 owner、关键路径、风险登记、证据缺口。方案推给 owner 确认，确认后由主 agent 建 Planner 任务并派发。

First Collection 后引导触发 Owner 确认后执行 任务计划 + Planner 任务

PMO Agent 日常

项目存续期间持续运行，维护项目认知和推进任务执行
5
日常 Source Collection
增量信息收集

按 Signal Mandate 扫描所有信源，只看上次扫描以来的新内容。产出候选列表但不自动入库——等日常资料 Review 时和 DRI 一起处理。

如果发现命中升级阈值的信号（Signal Mandate 定义的"立刻通知"条件），立即推给 DRI，不等 Review。

每天 2 次 自动执行 候选信息列表
6
日常资料 Review
认知对齐 + 素材入库

紧跟日常收集执行。和 DRI 对齐新发现的内容：更新认知对象（新的决策、风险、假设）、推断成员立场变化、计算认知差距。

DRI approve 后将新素材入库到 sources/。本质是 First Collection Phase 2+3 的增量版本。

每天 2 次（紧跟收集） 需要 DRI cognition/ + sources/
7
Project Heartbeat
项目运行脉搏

检查所有任务状态：没有 owner 的任务建议分配，超期任务催办，已提交的交付物评估结果。同步 Planner 和 SharePoint 状态，写心跳日志。

这是纯执行层面的推进——不做信息收集，不做认知对齐，只管"任务有没有在往前走"。

每小时 1 次 自动执行 催办 + 状态同步 + 日志
8
Task Review
战略层面的任务审视

每天早上回顾全局：有没有漏掉重要任务没规划到项目中？有没有过期或不再相关的任务需要清理？有没有从信号中发现"应该做但没人提出"的事？

和 Heartbeat 的区别：Heartbeat 管"已有任务有没有在推进"，Task Review 管"该做的事有没有变成任务"。

每天早上 1 次 发现后推给 DRI 新任务建议 + 清理
9
Weekly Learning
每周反思经验沉淀

回顾本周的心跳日志、review 记录、认知对齐记录，找到值得沉淀的 pattern：执行型教训（"这件事下次该怎么做"）和发现型洞察（"原来这个领域有这个规律"）。

和 DRI 确认后写入 learnings/。一周的信息量够发现 pattern——太早沉淀容易写出"正确但无用"的经验。

每周 1 次（周五下午） 需要 DRI 确认 learnings/

Member Agent 日常

每个项目成员的 Agent 独立运行，收集个人视角的信息并入库
10
Member Source Collection
成员视角的增量信息收集

和 PMO 的日常收集类似，但从成员个人的视角出发。扫描该成员的 Teams 消息、邮件、会议纪要等，按 Signal Mandate 判断和项目的相关性，产出候选列表。

不同成员看到的信息不同——一封邮件可能只有收件人的 Agent 才能看到。成员收集补充了 PMO 的盲区。

每天 2 次 自动执行 候选信息列表
11
Member Review
成员素材入库审批

紧跟成员收集执行。和成员确认哪些素材入库到项目的 sources/ 目录（按成员分子文件夹）。不做认知对齐——cognitive layer 由 PMO Agent 统一管理。

入库的素材对 Project Agent 和其他成员可见，Project Agent 负责去重和索引。同一信息出现在多个成员文件夹时，保留第一份完整内容，其余标注 duplicate。

每天 2 次（紧跟收集） 需要成员确认 sources/{member}/

信息流转

外部信号 → Source Collection (PMO + Member) ↓ 候选列表 → 资料 Review (DRI approve) ↓ 入库素材 → sources/ ↓ 认知推断 → Truth Gate (DRI 校准) → cognition/ ↓ 认知差距 → Member Alignment (成员确认) → stances/ ↓ 任务规划 → Planning / Task Review → Planner Tasks ↓ 任务执行 → Heartbeat (催办/验收) → deliverables/ ↓ 经验沉淀 → Weekly Learning (DRI 确认) → learnings/

设计原则

收集不入库

AI 扫描产出候选列表，但不自动写入 sources/。入库需要人类明确批准——这是隐私和相关性的门控。

AI 先亮模型

认知对象预填 candidate，成员立场预填 inferred。人类是校准者不是空白作者——AI 先提判断再让人确认。

扫描全面 呈现精炼

8 个信源全覆盖，但给用户看的是合并后的认知判断，不是 raw signal dump。

沉默不等于同意

成员没回复认知校准问题，标记 unknown，不标 aligned。Agent 不能把推断静默变成项目真相。

推进不需要人

Heartbeat 自动催办、验收、同步状态。不是每件事都要等 DRI——执行层面 agent 自主推进。

Signal 是证据 Object 是结论

signals/ 存外部事件（邮件、会议、新闻），objects/ 存项目认为是真的事。从证据到结论需要 DRI 确认。
